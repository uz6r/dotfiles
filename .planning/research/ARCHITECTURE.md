### Component Responsibilities

| Component | Responsibility | Typical Implementation |
|-----------|----------------|------------------------|
| `zsh/.zshrc` | Main shell config, platform detection, generic aliases | Single file with modular sourcing or split into `conf.d/` |
| `zsh/.zshrc.local` | Machine-specific overrides, secrets, company config | Gitignored, sourced last by `.zshrc` |
| `zsh/.p10k.zsh` | Powerlevel10k prompt theme config | Generated by `p10k configure`, committed |
| `git/.gitconfig` | Global git settings, aliases, ignores | Committed, uses `includeIf` for work repos |
| `nvim/.config/nvim/` | Neovim Lua config | Modular Lua files under `lua/` |
| `tmux/.tmux.conf` | Tmux keybindings, settings | Single file or split with `source-file` |
| `bin/` | Portable utility scripts | Shell scripts, no company-specific code |
| `scripts/` | Install/bootstrap helpers | Setup scripts, platform-specific installers |
| `Makefile` | Common tasks (install, test, lint) | Platform-aware targets |

### Structure Rationale

- **zsh/**: Contains all zsh-related config; `.zshrc.local` is gitignored so machine-specific overrides don't leak
- **bin/**: Only portable, generic scripts; company-specific scripts removed entirely
- **scripts/**: Installation and setup scripts separate from runtime utilities
- **One directory per tool**: Following the "one directory per tool" principle for clarity (source: shinyaz.com, 2026)

## Architectural Patterns

### Pattern 1: .zshrc.local for Machine-Specific Config

**What:** Keep a `.zshrc.local` file that is sourced at the end of `.zshrc` but never committed to the repository.

**When to use:** For machine-specific aliases, company-specific config, API keys, PATH additions that vary per machine.

**Trade-offs:** 
- ✅ Clean repo without sensitive or machine-specific config
- ✅ Easy to have different configs on different machines
- ✅ Safe place to put temporary experiments
- ⚠️ Need to document what typically goes in `.zshrc.local` for new machine setup

**Example:**
```zsh
# In .zshrc (committed):
# Load local overrides if they exist (always last)
if [ -f "$HOME/.zshrc.local" ]; then
  source "$HOME/.zshrc.local"
fi

# In .zshrc.local (gitignored, per-machine):
# Company-specific aliases (if still needed on work machine)
# alias core="cd $HOME/company/core"
# API keys
# export SOME_API_KEY="secret"
# Machine-specific PATH
# export PATH="$HOME/custom/bin:$PATH"
```

**Best Practice (from peternutter/dotfiles, brettdavies/dotfiles):** Provide a `.zshrc.local.example` template showing common patterns, but gitignore the actual `.zshrc.local`.

### Pattern 2: Modular .zshrc with Numeric Prefixes

**What:** Split `.zshrc` into multiple small files under a `conf.d/` directory with numeric prefixes to control load order.

**When to use:** When `.zshrc` grows beyond 200 lines or has distinct concerns (aliases, functions, tool configs).

**Trade-offs:**
- ✅ Each file has single responsibility
- ✅ Predictable load order via numeric prefixes
- ✅ Easy to disable modules (rename `.zsh` → `.zsh.disabled`)
- ✅ Clear separation of concerns
- ⚠️ Slightly more complex setup
- ⚠️ Need a loader loop in `.zshrc`

**Example:**
```zsh
# ~/.config/zsh/.zshrc (minimal loader)
ZDOTDIR="${ZDOTDIR:-$HOME/.config/zsh}"
for config in "$ZDOTDIR/conf.d/"*.zsh(N); do
  source "$config"
done

# ~/.config/zsh/conf.d/ structure:
# 00-exports.zsh      # Environment variables (loaded first)
# 10-path.zsh         # PATH modifications
# 20-functions.zsh    # Custom functions
# 30-aliases.zsh      # General aliases
# 40-tools.zsh        # Tool-specific (nvm, pyenv, etc.)
# 50-completions.zsh  # Completion settings
# 90-local.zsh        # Sources ~/.zshrc.local (loaded last)
# 99-prompt.zsh       # Prompt config (references everything above)
```

**Sources:** 
- ADR-083: Modular ZSH Configuration Architecture (docs.coditect.ai)
- Carmelyne Thompson, "2 Step Defense in Depth for Dotfiles" (2026)

### Pattern 3: Platform Detection with Existence Checks

**What:** Use `uname -s` for OS detection and `(( $+commands[tool] ))` or `[[ -x ... ]]` for feature detection instead of hardcoded paths.

**When to use:** Cross-platform dotfiles supporting macOS and Linux.

**Trade-offs:**
- ✅ No errors when tools aren't installed
- ✅ Works correctly on both macOS and Linux
- ✅ Handles both Apple Silicon and Intel Macs
- ⚠️ Slightly more verbose than hardcoded paths

**Example:**
```zsh
# Platform detection (from current .zshrc, good pattern)
export IS_MACOS=false
export IS_LINUX=false

case "$(uname -s)" in
  Darwin*) export IS_MACOS=true ;;
  Linux*) export IS_LINUX=true ;;
esac

is_darwin() { [[ "$IS_MACOS" == "true" ]]; }
is_linux() { [[ "$IS_LINUX" == "true" ]]; }

# Safe existence checks (from shinyaz.com, 2026)
if is_darwin; then
  alias ll='ls -lahG'
else
  alias ll='ls -lah --color=auto'
fi

# Check if command exists before using
if (( $+commands[bat] )); then
  alias cat='bat'
fi

# Check if file is executable
if [[ -x "/opt/homebrew/bin/brew" ]]; then
  export HOMEBREW_PREFIX="/opt/homebrew"
fi
```

## Scaling Considerations

| Scale | Architecture Adjustments |
|-------|--------------------------|
| 1 machine | Current setup is fine, simple `.zshrc` + `.zshrc.local` |
| 2-5 machines | Modular `conf.d/` structure helps manage per-machine variations |
| 5+ machines | Consider chezmoi for templating, or keep using stow with `.zshrc.local` per machine |
| Team sharing | Remove ALL personal preferences, use `.zshrc.local` for individual tweaks |

### Scaling Priorities

1. **First bottleneck:** Managing per-machine differences → solved by `.zshrc.local` pattern
2. **Second bottleneck:** Multiple tools with conflicting configs → solved by modular `conf.d/` structure

## Anti-Patterns

### Anti-Pattern 1: Company Config in Main .zshrc

**What people do:** Keep company-specific aliases, functions, and PATH entries directly in `.zshrc`.

**Why it's wrong:** 
- Breaks when leaving the company (need to audit entire file)
- Makes the repo non-portable (can't share, confusing for others)
- Increases cognitive load (irrelevant config always visible)

**Do this instead:** 
- Put company config in `.zshrc.local` (gitignored, per-machine)
- Or remove entirely if leaving (current cleanup task)
- Keep `.zshrc` generic and portable

**Current state:** `.zshrc` has `localdev()` function (lines 153-184) and commented Courtsite aliases (lines 314-328). These should be removed. Good news: `.zshrc.local` already has the Courtsite aliases separated.

### Anti-Pattern 2: Hardcoded Platform-Specific Paths

**What people do:** Write `/usr/local/bin` (Intel Mac) or `/opt/homebrew/bin` (Apple Silicon) directly in config.

**Why it's wrong:** Breaks on different architectures or when tools are installed elsewhere.

**Do this instead:** 
```zsh
# Good: Auto-detect Homebrew prefix
if is_darwin; then
  if [[ "$(uname -m)" == "arm64" ]]; then
    export HOMEBREW_PREFIX="/opt/homebrew"  # Apple Silicon
  else
    export HOMEBREW_PREFIX="/usr/local"      # Intel
  fi
elif command -v brew >/dev/null 2>&1; then
  export HOMEBREW_PREFIX="$(brew --prefix)"  # Linuxbrew
fi
```

**Current state:** The existing `.zshrc` already does this correctly (lines 60-77). Keep this pattern.

### Anti-Pattern 3: Company Scripts in bin/

**What people do:** Keep company-specific utility scripts (like `sinar-pi-setup`) in the shared `bin/` directory.

**Why it's wrong:** 
- `bin/` should contain portable, generic utilities
- Company scripts become dead code after leaving
- Clutters the utility namespace

**Do this instead:** 
- Delete company-specific scripts entirely during cleanup
- Keep only generic, portable scripts (like `profile-zsh`, `audit-nvim-plugins`)
- If company scripts are needed on work machines, keep them in a separate, gitignored location

**Current state:** `bin/sinar-pi-setup` and `bin/sinar-pi-wifi-setup` are Courtsite-specific. They should be deleted.

## Integration Points

### External Services

| Service | Integration Pattern | Notes |
|---------|---------------------|-------|
| Homebrew | Auto-detect prefix, add to PATH | Works on macOS and Linux |
| Oh-My-Zsh | Clone to `~/.oh-my-zsh`, source `$ZSH/oh-my-zsh.sh` | Plugins: git, sudo, web-search, etc. |

| fzf | Source `~/.fzf.zsh` if exists | Optional, auto-detect |
| nvm | Source `$NVM_DIR/nvm.sh` if exists | Optional, lazy-loaded |

### Internal Boundaries

| Boundary | Communication | Notes |
|----------|---------------|-------|
| `.zshrc` ↔ `.zshrc.local` | Sourced at end of `.zshrc` | `.zshrc.local` can override anything |
| `bin/` scripts ↔ PATH | Added to PATH in `.zshrc` | Scripts must be portable (no company refs) |
| Makefile ↔ Install scripts | Make targets call scripts | Scripts handle platform detection |
